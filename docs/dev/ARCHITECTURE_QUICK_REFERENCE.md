## 📊 核心对比

### 数据模型

```
┌─ V1 (一对一强绑定) ──────────────────┐
│                                       │
│  vault_items                          │
│  ├─ id                                │
│  ├─ title                             │
│  ├─ credentials                       │
│  └─ project_id (FK) ← 必须            │
│                                       │
│  projects (可以为空)                  │
│  ├─ id                                │
│  └─ name                              │
│                                       │
└───────────────────────────────────────┘

❌ 问题: 凭证必须关联，无法独立存在
❌ 无法跨项目共享凭证
❌ 导入功能混乱
```

```
┌─ V2 (多对多灵活关联) ─────────────────┐
│                                       │
│  vault_items ✅ 独立                  │
│  ├─ id                                │
│  ├─ title                             │
│  ├─ credentials                       │
│  └─ (无project_id)                    │
│                                       │
│  projects ✅ 独立                     │
│  ├─ id                                │
│  ├─ name                              │
│  └─ (无凭证依赖)                      │
│            ↓ N:N 关联 ↓               │
│  credential_project_relations ✅      │
│  ├─ credential_id                     │
│  ├─ project_id                        │
│  └─ relationship_type                 │
│                                       │
│  chrome_imported_passwords ✅         │
│  ├─ import info                       │
│  └─ optional vault_item_id            │
│                                       │
└───────────────────────────────────────┘

✅ 完全独立，最大灵活性
✅ 支持N:N关联
✅ 完整导入历史管理
```

## 🔄 关键场景对比

### 场景 1: 添加独立的 API KEY

**V1 (旧方式) - 不推荐** ❌
```
问题: API KEY 必须关联某个项目
解决: 创建虚拟项目 "通用 API KEY"  ← 不优雅

导致:
- 许多虚拟项目
- UI 混乱
- 查询性能下降
```

**V2 (新方式) - 推荐** ✅
```
1. 创建凭证 (type: 'api-key', title: 'Aliyun Access Key')
2. 不关联任何项目
3. 在 API KEY 管理视图中显示
4. 需要时关联到项目

优点:
- 无需虚拟项目
- 完全独立管理
- UI 清晰明了
```

---

### 场景 2: 跨项目共享数据库凭证

**V1 (旧方式)** ❌
```
需求: 项目A和项目B共同使用同一个MySQL数据库

解决方案1: 复制凭证
- 创建两个相同的凭证
- 问题: 修改时需要改两份
- 不一致风险高

解决方案2: 创建共享项目
- 在共享项目中存储凭证
- 项目A和B中引用共享项目
- 复杂且易错
```

**V2 (新方式)** ✅
```
需求: 项目A和项目B共同使用同一个MySQL数据库

解决方案: 一个凭证，关联多个项目
1. 创建凭证 (title: '生产主库', type: 'database')
2. 关联到项目A (relationship_type: 'associated')
3. 关联到项目B (relationship_type: 'associated')
4. 修改时自动生效

优点:
- 单一事实来源
- 修改一次生效处处
- 关系类型灵活标记
```

---

### 场景 3: Chrome 导入和冲突处理

**V1 (旧方式)** ❌
```
流程:
1. 导入Chrome密码 → 直接创建凭证
2. 冲突时 → 需要手动删除和重建
3. 导入历史 → 无法追踪 (混在凭证里)

问题:
- 无法看到导入源
- 无法追踪哪些被导入
- 无法回滚
```

**V2 (新方式)** ✅
```
流程:
1. 导入Chrome密码 → 存入chrome_imported_passwords表
2. 显示导入预览和冲突分析
3. 用户确认 → 创建vault_item并关联import记录
4. 完整的导入历史和统计

优点:
- 完整的导入历史
- 冲突检测和处理
- 可追踪和可审计
- 支持多次导入
```

## 📑 表结构对比

### V1 表结构
```
vault_items
├─ DEFAULT: id, title, category, url
├─ ADDED:   type, host, port, metadata...
└─ FK:      project_id → projects (强制)

projects
├─ id, name, description...
└─ 与凭证的关系: 一对多 (一个项目对应多个凭证)
```

### V2 表结构
```
vault_items (与V1类似，仅移除project_id)
├─ DEFAULT: id, title, category, url
├─ ADDED:   type, host, port, metadata...
└─ FK:      (无project_id)

projects (新增)
├─ id, name, description...
└─ 与凭证的关系: 多对多

credential_project_relations (新增)
├─ 支持N:N关联
└─ 支持关系类型标记 (primary/associated/shared)

chrome_imported_passwords (新增)
├─ 管理导入历史
└─ 可选关联到vault_item

api_keys_registry (新增, 可选)
├─ 独立管理API KEY
└─ 支持过期和轮转
```

## 🎨 UI 变化

### V1 UI 结构
```
Sidebar:
├─ 项目 A
│  └─ 凭证列表 (20个)
├─ 项目 B
│  └─ 凭证列表 (15个)
└─ 项目 C
   └─ 凭证列表 (10个)

问题: 只能按项目浏览凭证
```

### V2 UI 结构  
```
Sidebar:
├─ 📚 凭证库
│  ├─ 所有凭证 (45个)
│  ├─ 未归档凭证 (43个)
│  ├─ 未关联凭证 (5个)  ← 新：独立凭证
│  └─ 按类型过滤
│
├─ 🏢 项目管理
│  ├─ 项目 A (20个凭证)
│  ├─ 项目 B (15个凭证)
│  └─ 项目 C (10个凭证)
│
├─ 📥 导入和同步
│  ├─ Chrome 导入历史
│  └─ 冲突处理
│
└─ 🔑 API KEY 管理 (可选)
   ├─ Aliyun Keys
   ├─ AWS Keys
   └─ 其他服务

优势: 多维度浏览，完整功能
```
